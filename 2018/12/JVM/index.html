JVM Memory

运行时内存模型

  • 程序计数器 （Program Counter Register）：当前线程所执行的字节码的行号指示器。为了线程切换后能恢复到正确的执行位置，每个线程又自己的独立的程序计数器。如果正在执行一个Java方法，它指向正在执行的字节码指令的地址。如果正在执行一个Native方法，计数器为空。
  • 虚拟机栈 （VM Stack）：线程私有，生命周期与线程相同。每个方法（Method）被执行的时候都会创建一个栈帧（Frame），每个方法被调用到执行完成，就对应着一个栈帧从入栈到出栈的过程。
    • 栈帧存储局部变量表，操作栈，动态链接，方法出口等等信息。
    • 局部变量表保存编译期可知的各种基本数据类型，对象引用。其所需的内存空间在编译期完成分配。
    • 栈深度过深时抛出StackOverflowException，内存过大时抛出OutOfMemoryException。
  • 本地方法栈 （Native Stack）
    • 类似与虚拟机栈，但是为本地方法栈服务。
  • 方法区 （Method Area）
    • 各个线程共享，存储已被虚拟机加载的类信息，常量，静态变量等
    • 在HotSpot虚拟机上用永久代实现方法区，但并不是所有的JVM都是这样实现。
  • 堆 （Heap）
    • 被所有线程共享的区域，虚拟机启动时创建。存放所有对象。
    • 可以物理不连续。
  • 直接内存（Direct Memory）
    • nio库会在堆外用Native函数直接分配内存，然后通过Java堆中的DirectByteBuffer对象作为引用来操作。
    • 这部分内存不属于Java堆的限制，但是也是需要管理的。

对象访问

上面这句话中new Object()会反应在Java Heap中，开辟一块地址给这个对象，但是对象类型数据（对象类型，父类，接口，方法等存在方法区）。 Object obj会反映在VM Stack中的局部变量表中，作为一个Reference类型出现，主流访问对象的方式有两种：句柄（Handler）和直接指针。

  • 句柄：Java堆中划分一块地址给句柄池，reference指向句柄池中的地址，句柄中包含了对象实例数据和类型数据。
    • 优点：reference存储的地址很‘稳定’。在对象被移动的时候（GC中经常出现），reference不需要改变，只需要改变句柄中的地址。

  • 直接指针：reference直接存储对象实例地址。
    • 优点：速度更快，节省一次指针定位的时间开销。

垃圾收集

垃圾收集基本主要集中在Java Heap中，其他都比较固定。首先要能分辨对象是否已经“死去”。

对象已死？

引用计数

给对象添加一个引用计数器，有一个地方引用的时候+1，不再引用的时候-1，计数器为0的时候即为不可能再被使用的。缺点：很难解决对象之间互相引用的问题。

根搜索计数

同过一系列的名为：GC Roots的对象作为节点，从节点处向下搜索，Reference Chain为走过的路径。当一个对象不可达的时候，则这个对象是可以被回收的。

在Java中，GC Roots包括：

  • 虚拟机栈中的引用对象
  • 方法区中类静态变量
  • 方法区中常量
  • JNI中引用的对象

强引用还不够

我们希望表述这样的行为：当我们内存足够的时候，我们希望保留这个对象，如果内存紧张，把它回收了也没关系。主要用在缓存领域。

  • 强引用（Strong Reference）：最常见，只要有强引用存在垃圾回收器就不能回收该对象。
  • 软引用（Soft Reference）：还有用，但是非必须。系统将要发生内存溢出之前，会把这些对象列入回收范围。
  • 弱引用（Weak Reference）：非必需对象，强度比弱引用更低，弱引用的对象只能存活到下一次垃圾收集之前，无论是否内存足够。
  • 虚引用（Phantom Reference）：最弱的一种引用，完全不会对其生命周期构成影响。为一个对象设置虚引用的唯一理由就是希望在对象被回收时收到一个通知。

回收方法区

方法区一般比较稳定，回收的性价比较低，主要集中在回收废弃常量和无用的类。

垃圾回收算法

标记-清除算法

分为标记，清除两个阶段，首先标记出所有需要回收的对象，然后统一回收掉全部对象。
主要问题有两个：1）标记和清除效率都不高。2）空间问题：清除后会留下大量不连续的内存碎片。

标记-整理算法

是标记清理的改进，标记过程相同，在整理步骤中，让所有存活的对象都像一段移动，直接清理掉边界以外的内存。

复制算法

把内存分为两部分，每次只使用一块，当这一块的内存用完以后，把活着的对象复制到另一块内存中，接下来只使用另一块。然后直接清理掉已使用过的内存空间。代价就是可用内存减少。

当前主流实现是把内存分为3部分，一块为较大的Eden空间，和两块较小的Survivor空间。每次使用一块Eden空间和一个Survivor空间，回收时，将Eden空间和Survivor空间中活着的对象拷贝到另一块Survivor中，并清理原Eden空间和Survivor空间。

虽然大部分时候Survivor空间足够用，但是总是有例外的情况，所以必须要用额外的空间（老年代）担保。

垃圾收集器

收集器有很多种，各自有自己的优点和缺点。

  • 新生代
    • Serial
    • ParNew
    • Parallel Scavenge
  • 老年代
    • CMS
    • Parallel Old
    • Serial Old
  • 通用
    • G1
    • ZGC

Serial

在新生代中，是一个单线程的收集器，使用复制算法，在收集的时候必须要stop the world。也导致了早期Java慢，卡顿的坏印象。但是它简单而高效。在当前Client端设备中，虚拟机内存一遍不会很大，停顿时间几乎感觉不到，所以也是很好用的。

还有ParNew是Serial在新生代的多线程版本，但是仍然需要Stop The World。是Server模式下的首选，因为当前只有ParNew可以和CMS配合。

在老年代中，使用标记整理算法。可以搭配Serial新生代使用，也可以做CMS的备份预案。

Parallel

Parallel Scavenge是一个新生代收集器，使用复制算法，并行。目的是达到可控制的吞吐量（throughput）。吞吐量=运行用户代码时间/(总时间)。吞吐量更高可以提高CPU效率。适合后台计算，不涉及前台交互。

  • MaxGCPauseMills：控制GC停顿时间，牺牲吞吐量和新生代空间大小。
  • GCTimeRatio：0~100，吞吐量的倒数。越大，垃圾回收时间越少。

在老年代叫Parallel Old。使用标记整理算法和多线程。

CMS（Concurrent Mark Swap）收集器

以获取最短回收停顿时间为目标。在重视服务速度的时候使用。使用标记清除算法。整个回收过程包括：

  1. 初始标记（STW）仅仅标记GC Roots能直接关联到的对象。
  2. 并发标记。GC Roots Tracing
  3. 重新标记（STW）修正并发标记期间，因为用户程序继续运行而导致的标记变动的部分。
  4. 并发清除

缺点：

  • 对CPU资源要求更高
  • 无法处理浮动垃圾（在并发标记时产生的垃圾），如果CMS预留的内存不够用，会出现Concurrent Mode Failure，虚拟机将采用后备预案Serial Old进行一次Full GC。
  • 清除阶段不整理，将会出现大量的空间碎片。

G1（Garbage First）收集器

基于标记整理算法，不会产生空间碎片。可以精确的控制停顿。可以在基本不牺牲吞吐量的前提下完成低停顿的回收。G1将整个Java堆分成多个大小固定的独立区域，跟踪区域里垃圾堆积程度，优先回收垃圾最多的区域。

How to choose

  • If the application has a small data set (up to approximately 100 MB), then select the serial collector with the option -XX:+UseSerialGC.
  • If the application will be run on a single processor and there are no pause time requirements, then let the VM select the collector, or select the serial collector with the option -XX:+UseSerialGC.
  • If (a) peak application performance is the first priority and (b) there are no pause time requirements or pauses of 1 second or longer are acceptable, then let the VM select the collector, or select the parallel collector with -XX:+UseParallelGC.
  • If response time is more important than overall throughput and garbage collection pauses must be kept shorter than approximately 1 second, then select the concurrent collector with -XX:+UseConcMarkSweepGC or -XX:+UseG1GC.

内存分配策略

  • 对象优先在Eden空间中分配，如果Eden空间容量不足，则触发一次Minor GC（只做新生代的GC），Minor GC发现Survivor空间盛不下的时候会透过担保机制将本应该拷贝到Survivor空间的对象转移到老年代中去。
  • 大对象直接分配到老年代，避免发生大量的Survivor空间的复制。
  • 长期存活的对象会进入到老年代。
  • 动态年龄判定。
  • 空间分配担保，采用之前每次晋升到老年代的平均大小作为阈值，如果小于老年代的剩余空间，则需要进行一次Full GC。

如何调优

现在硬件提升，内存越来越大，JVM可以使用的内存可以超过10G。但是要为JVM配备大内存，就要能保证控制Full GC的频率不能太高。至少要高于10几个小时，能在深夜自动出发Full GC最好。要点就是不要出现批量的，长时间生存的大对象，才能保证老年代稳定。

虚拟机子系统

虚拟机类加载

加载，验证，准备，初始化会按部就班的开始。但是通常是交叉进行，在一个过程执行中调用另一个过程。

加载（Loading）

通过类加载器加载类。

  • 通过类的全限定名获取类的二进制字节流
  • 转换成方法区的运行时数据结构
  • Java堆中生成Class对象

类加载器

类加载器虽然只用于实现类的加载动作，但是却影响深远。任何一个类，如果要比较唯一性，都需要在同一个加载器的前提下才有意义。包括equals, isInstance, instance of。

双亲委派模型

大体上说，系统提供了三种类加载器

  • 启动类加载器（Bootstrap ClassLoader），仅加载\lib目录下的某些jar
  • 扩展类加载器（Extension ClassLoader），加载\lib\ext目录
  • 应用程序加载器（Application ClassLoader），一般默认的类加载器

双亲委派模型要求除了顶层的启动类加载器外，其他的都需要有自己的父类加载器。一般采用组合关系来复用父加载器的代码。

工作过程：首先，把类加载请求分配给父类加载器，每一个层次的类加载器都是这样，因而最后所有的加载请求最终都传送到顶层的启动类加载器中，只有当父加载器无法完成这个加载请求的时候，子类加载器才会尝试自己去加载。保证顶层类只包含一个版本。

初始化（Initialization）

立刻初始化的四种情况

  • 遇到字节码new, getstatic, putstatic, invokestatic。如new创建新对象，读取设置一个静态字段（被final修饰的除外），调用一个类的静态方法。
  • reflect反射
  • 触发初始化的时候，要保证父类已经被初始化了
  • main主类

详细

执行类构造器clinit方法。

  • clinit方法是编译器收集类中所有类变量的赋值动作和静态语句块的语句合并而成的。
  • 父类的clinit一定早于子类的clinit方法
  • JVM会保证在多线程中被正确的加锁和同步，所以不要放耗时很长的工作，这样的阻塞很难被发现。

字节码执行引擎

在概念上，虚拟机的执行引擎都是读入字节码，处理字节码，输出执行结果。虚拟机中可能包含多个执行引擎，比如解释器，编译器。

栈帧是虚拟机栈中的个体。存储了局部变量表，操作数栈，动态链接和方法返回地址等。每个方法的栈帧的具体内存在编译的时候就确定了。具体如下图

局部变量表

在方法执行时，如果是实例方法，局部变量表中的第0位索引的 Slot 默认是用于传递方法所属对象实例的引用，可使用 this 来访问这个隐含参数。其余参数按照参数表的顺序来排列。参数表分配完毕后，根据方法体内定义的变量顺序和作用域分配其余的 slot。

局部变量表中的 Slot 是可以重用的，如果一个变量的作用域不一定会覆盖整个方法体，则可以重用该变量。

操作数栈

后入先出的栈。最大深度也是固定的。算数运算，参数传递都可以通过操作数栈来事先。可以通过两个栈帧重叠一部分，来进行参数传递，避免额外的复制。

方法调用

方法调用不等于方法执行，唯一任务就是确定被调用方法的版本。由于 Class 文件在编译过程不包含连接步骤，一切方法调用在 Class 文件里面存储的都只是符号引用，而不是方法在实际运行时内存布局的入口地址。

解析

在类加载的解析阶段，会将 Class 文件中的一部分符号引用转化成直接引用。前提为：调用方法在编译时就已经能确定下来的。在 Java 语言中，主要有静态方法和私有方法两类。

分派（Dispatch）

分派可能是动态的也可能是静态的。方法的接受者和方法的参数统称为方法的宗量。根据分派的宗量数可分为单分派与多分派。

  • 静态分派
    • 主要处理重载问题。
    • 变量的定义是静态类型（外观类型），实际的对象为实际类型。静态类型编译期就可知，而实际类型只有在运行期才能确定。
  • 动态分派
    • 主要处理重写问题
    • invokevirtual 指令比较聪明
      • 找到操作数栈顶的第一个元素所指向的对象的实际类型，记做 C
      • 如果在 C 中找到相匹配的方法，则做权限校验，通过返回这个方法的直接引用
      • 否则，按照继承关系向上寻找
    • 可以把同样的静态类型定位到不同的方法中，而这个行为只能在运行时才能确定。
  • 分派
    • 静态分派过程中，一看静态类型，二看方法参数，所以是静态多分派
    • 动态分派过程中，只看实际类型，所以动态单分派

程序编译与优化

早期优化（编译期）

我们限制编译期为从 java 文件转变成 class 文件的过程。Javac 对这类代码几乎没有任何优化措施，但是提供了很多语法糖。编译过程大致包括是三部分

  • 解析与填充符号表的过程
  • 插入式注解处理器的注解处理过程
  • 分析与字节码生成过程

  • 词法分析：字符流转变为 Token 的集合
  • 语法分析：根据 Token 序列构造抽象语法树
  • 填充符号表：符号地址和符号信息的 KV 对
  • 语义分析：对结构正确的源程序进行上下文有关性质的审查
  • 解语法糖
  • 字节码生成

晚期优化（运行期）

解释器可以使程序迅速启动和执行，省去编译的时间。当程序运行后，随着时间的推移，编译器逐渐发挥作用，把越来越多的代码编译成本地代码，获得更高的执行效率。解释器还可以作为编译激进优化的逃生门。

热点代码有两类：

  1. 被多次调用的方法
  2. 被多次执行的循环体

热点探测判断方式：

  1. 基于采样的热点探测
  2. 基于计数器的热点探测
    1. 方法调用计数器（Invocation Counter）
    2. 回边计数器（Back Edge Counter）

常见优化方式

  • 公共子表达式消除
  • 冗余访问消除
  • 无用代码消除
  • 数组范围检查
  • 方法内联
    • 去除调用方法的成本
    • 为其他优化建立基础
    • 许多情况下内联都是激进优化
  • 逃逸分析
    • 为其他优化提供依据
    • 目的就是发现不会逃逸到线程/方法外的变量
    • 栈上分配
    • 同步消除
    • 标量替换

最主要的一个技术难点就是Java的分离编译（separate compilation）和动态类加载（dynamic class loading）/动态链接（dynamic linking）。不知道运行时会加载并链接上什么代码。Java 大量的虚方法，让静态分析无法保证当前看到的方法就是运行时需要调用的方法。

内存模型和线程

内存模型

主内存：虚拟机内存的一部分，所有变量都存储在主内存中。
工作内存：保存了该线程所使用到的变量的主内存副本拷贝。线程对变量的所有操作都必须在工作内存中进行，而不能直接读写主内存中的变量。不同线程之间也无法直接访问对方工作内存中的变量。

volatile

JVM 提供的最轻量级的同步机制，

volatile 提供两个特性：

  • 此变量对所有线程的可见性
    • 对volatile的所有写操作都会立刻反应到其他线程之中。
    • 新值立刻同步到主内存中，每次使用前从主内存刷新
  • 禁止指令重排序优化
    • 普通的变量只能保证执行过程中所有依赖赋值的地方都能获得正确的结果，不能保证赋值操作的顺序与程序代码执行顺序一致。
    • volatile 能保证执行到该行前所有语句已经执行完毕。

原子性、可见性与有序性

原子性：JVM 保证基本类型读写具有原子性（long 和 double），更大范围的原子性可以使用 synchronized 关键字。
可见性：当一个线程修改了共享变量的值，其他线程能立刻得知这个修改。volatile 和 synchronized 能实现可见性。
有序性：Java天然的有序性可以总结为一句话：如果在本线程内观察，所有的操作都是有序的（线程内表现为串行的语义），如果在另一个线程中观察，所有的操作都是无序的。volatile 和 synchronized 能实现有序性。

线程安全与锁优化

Java 采用了抢占式调度，每个线程由系统分配执行时间。
Java 定义了5种进程状态，分别为：

  • 新建（New）：创建后尚未启动的线程
  • 运行（Runnable）：线程可能正在执行，也可能正在等待 CPU 为它分配执行时间。
  • 无限期等待（Waiting）：不会被分配 CPU 时间，要等待被其他线程显式唤醒
  • 限期等待（Timed Waiting）
  • 阻塞（Blocked）

线程安全的实现方法

互斥同步

  • synchronized
    • 在进入时，首先尝试获取对象的锁
    • 如果对象没有被锁定，或者当前线程已经拥有了那个对象的锁，把锁计数器加1
    • 退出时把锁减1
    • 计数器为0时释放锁
    • 如果获取锁失败了，则当前线程要阻塞等待
    • Java 的线程是映射到操作系统的原生线程之上的，阻塞或者唤醒线程都需要从用户态转变到核心态。
  • java.util.concurrent.ReentrantLock
    • 基本功能和 synchronized 相同
    • 高级功能包括
      • 等待可中断
      • 公平锁
      • 绑定多个 condition 对象（条件）

非阻塞同步

互斥同步是阻塞同步，是悲观的并发策略，总是认为只要不去做正确的同步措施，肯定会出问题。
而随着硬件指令集的发展，我们有了：基于冲突检测的乐观并发策略。先进行操作，如果出现了冲突，那再进行其他的补偿措施。这种同步操作属于非阻塞同步。

需要原子操作：CAS 比较并交换

无同步方案

  1. 可重入代码（Reentrant Code），所需要的参数只从方法的参数列表中来，并且不调用非可重入的方法，不共享资源。
  2. 线程本地储存（Thread Local Storage），每个线程存变量的一份拷贝，线程之间实际不共享变量。

锁优化

自旋锁与自适应自旋

许多应用中，共享数据的锁定状态只会持续很短的一段时间，为了这段时间挂起和恢复线程并不值得。我们可以让后面请求锁的线程执行一个忙循环（自旋）等一会儿，但是并不放弃处理器的执行时间。如果锁占用时间过长，还是要回到阻塞状态。所以如果很不幸，那我们付出的代价会更大。

自适应自旋锁是一个优化，自旋的时间不再固定，而是由前一次在同一个锁上的自旋时间及锁的持有者的状态来决定。在同一个锁对象上，自旋等待刚刚获得成功，并且持有锁的线程正在运行中，那这次自旋也很有可能再次成功，它允许自旋等待更久的时间。

锁消除

如果能检测到一段同步代码不可能存在共享数据竞争，那 JIT 会把锁清除。

锁粗化

如果一系列的连续操作都对同一个对象反复加锁解锁，那 JVM 会酌情把锁粗化，只加一次锁。

轻量级锁

并不能代替重量级锁（synchronized），本意是在没有多线程竞争的前提下，减少传统的重量级锁使用操作系统互斥量产生的性能损耗。

代码进入同步块的时候，如果此同步对象没有被锁定，则尝试将对象的 Mark Word 更新为指向 Lock Record 的指针，如果更新成功，则轻量锁获取成功。如果更新操作失败，则会检测当前对象的 Mark Word 是否指向当前线程的栈帧，如果是，则说明当前线程已经拥有了这个对象的锁，可以进入同步块继续执行。否则说明这个锁对象已经被别的线程抢占了，要膨胀成重量锁。

偏向锁

偏向锁就是在无竞争的情况下，把整个同步都消除掉。锁对象第一次被线程获取的时候，虚拟机把锁对象头中的标志位设为偏向模式，同时做 CAS 操作记录线程 ID。如果操作成功，以后持有偏向锁的线程每次进入该同步块，都不需要进行任何同步操作。

当另一个线程去获取这个锁的时候，偏向模式宣告结束。
