View Model

  • Logical view: System Architect’s View. The logical view is concerned with the functionality that the system provides to end-users. The system is decomposed into object classes. UML diagrams used to represent the logical view include, use case diagram, layered diagram, package diagram.
  • **Development View: ** Implementation View. Subsystem decomposition
    • an enumeration of all subsystems in the implementation model.
    • component diagrams illustrating how subsystems are organized in layers and hierarchies
  • Process View: The process view deals with the dynamic aspects of the system, explains the system processes and how they communicate, and focuses on the runtime behavior of the system. Sequence Diagram
  • Deployment View: Physical View. It is concerned with the topology of software components on the physical layer as well as the physical connections between these components.

Architecutural Style

Architectural Style: An Architectural Style defines a set of rules that describe, the way in which the components interact

Object-Oriented Style

Components are objects: Data and associated operations
Connectors are messages and method invocations

Objects are responsible for their internal representation integrity
Internal representationis hidden from other objects

Layered Style

Hierarchical system organization. Each layer exposes an interface (API) to be used by above layers.

Each layer acts as a

  • Server: service provider to layers “above”
  • Client: service consumer of layer(s) “below”

Connectors are protocols of layer interaction

Changes in a layer affect at most the adjacent two layers. Different implementations of layer are allowed as long as interface is preserved.

Performance

Client-Server Style

Components are clients and servers
Connectors are RPC-based network interaction protocol

Servers do not know number or identities of clients

3-Tier

Peer-to-Peer Style

State and behavior are distributed among peers which can act as either clients or servers.

Peers: independent components, having their own state and control thread.
Connectors: Network protocols, often custom.

Highly robust in the face of failure of any given node.

Batch Sequential Style

Separate programs are executed in order; data is passed as an aggregate from one program to the next.

Pipe and Filter Style

Components are filters
Connectors are pipes

Filters are independent (no shared state)
Filter has no knowledge of up- or down-stream filters

Filter addition,replacement, and reuse
Batch organization of processing

Blockboard Style

Two kinds of components
Central data structure — blackboard
Components operating on the blackboard

System control is entirely driven by the blackboard state.

Rule-Based Style

Inference engine parses user input and determines whether it is a fact/rule or a query. If it is a fact/rule, it adds this entry to the knowledge base. Otherwise, it queries the knowledge base for applicable rules and attempts to resolve the query.

Behavior of the application can be very easily modified through addition or deletion of rules from the knowledge base.

Interpreter Style

Interpreter parses and executes input commands, updating the state maintained by the interpreter.

Components: Command interpreter, program/interpreter state, user interface. Connectors: Typically very closely bound with direct procedure calls and shared state.

Mobile-Code Style

Components: “Execution dock”, which handles receipt of code and state; code compiler/interpreter
Connectors: Network protocols and elements for packaging code and data for transmission.

MapReduce

Implicit Invocation Style

Event announcement instead of method invocation
“Listeners” register interest in and associate methods with events.
System invokes all registered method simplicitly

No assumption about processing in response to events

Publish-Subscribe

Subscribers register/deregister to receive specific messages or specific content. Publishers broadcast messages to subscribers either synchronously or asynchronously.

Components: Publishers, subscribers, proxies for managing distribution
Connectors: Typically a network protocol is required.

Event-Based Style

Independent components asynchronously emit and receive events communicated over event buses.

Components: Independent, concurrent event generators and/or consumers
Connectors: Event buses (at least one)

#ATAM

To assess the consequences of architectural decisions in light of quality attribute requirements.
Human-centric process for identifying risks early on in software design.

  • Scenario – A scenario is a short statement describing an interaction of one of the stakeholders with the system
  • Stakeholder – An individual, team, or organization (or classes thereof) with interests in, or concerns relative to, a system
  • Architectural view – A representation of a whole system from the perspective of a related set of concerns
  • Functional requirements - specify what software has to do.
  • Non-functional requirements specify how well it should be done.

ATAM Business Drivers

The system’s critical functionality
Any technical, managerial, economic, or political constraints
The project’s business goals and context
The principal quality attribute (NFP) goals

ATAM Scenarios

  • Use-case scenarios : Describe how the system is envisioned by the stakeholders to be used
  • Growth scenarios : Describe planned and envisioned modifications to the architecture
  • Exploratory scenarios : Try to establish the limits of architecture’s adaptability

ATAM Analysis

Objective is to establish relationship between architectural approaches and quality attributes

System architects and ATAM evaluation team work together to answer these questions and identify

  • Risks
  • Non-Risks
  • Sensitivity points
  • Trade-off points

Steps

  1. Present ATAM (Overview)
  2. Present Business Drivers
    1. Business context for the system
    2. High-level functional requirements
    3. High-level quality attribute requirements
  3. Present the Architecture
    1. Technical constraints
    2. other systems interact
    3. architecture approaches/style
  4. identify architectural approaches
  5. generate utility tree
    1. Identify, prioritize, and refine the most important quality attribute goals by building a utility tree.
    2. Select the most important quality goals to be the high-level nodes (typically performance, modifiability, security, and availability)
  6. Analyze Architectural Approaches
    1. Sensitivity
    2. Tradeoffs
    3. Risk
    4. Non Risk
  7. Brainstorm & Prioritize Scenarios
  8. Analyze Architectural Approaches
  9. Present ATAM results

Microservice

Problems of Software Monoliths

A Software Monolith is One build and deployment unit, One codebase, One technology stack
Huge and intimidating code base for developers

  • Development tools get overburdened
    • refactoring take minutes
    • builds take hours
    • testing in continuous integration takes days
  • Scaling is limited
    • Running a copy of the whole system is resource-intense
    • It doesn’t scale with the data volume out-of-the-box
  • Deployment frequency is limited
    • Re-deploying means halting the whole system
    • Re-deployments will fail and increase the perceived risk of deployment

Principles of Microservices

  • Rely only on the public API
  • Use the right tool for the job
  • Secure your services
  • Be a good citizen within the ecosystem
  • More than just technology transformation
  • Automate everything

What is microservice? Why popular now? Comparison with SOA?

A microservice architecture puts each element of functionality into a separate service…

SOA systems also focus on functional decomposition,but

  • services are not required to be self-contained with data and UI, most of the time the contrary is pictured.
  • It is often thought as decomposition within tiers, and introducing another tier – the service orchestration tier

In comparison to microservice

  • SOA is focused on enabling business-level programming through business processing engines and languages such as BPEL and BPMN
  • SOA does not focus on independent deployment units and its consequences
  • Microservice can be seen as “SOA –the good parts”

What is devOps

software development method that stresses communication, collaboration and integration between software developers and information technology(IT).

software developers manage code, information technology manage machine.

Principles of DevOps

Develop and test in an environment similar to production
Deploy builds frequently
Validate operation quality continuously
